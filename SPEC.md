# xdotter 规范

## 状态

本文档是 xdotter 用户可见行为、程序逻辑和交互语义的事实来源。

## 规范边界

本文档定义 xdotter 的语言无关、实现无关的用户可见行为契约。

本文档约束外部可观察行为，包括配置语义、命令交互、文件系统影响、错误分类、退出码、stdout/stderr 边界、安全不变量和必要性能边界。

本文档不约束实现语言、库、框架、内部模块划分、数据结构、测试组织或逐字输出文案，除非这些细节会影响用户可见行为、安全语义或跨实现兼容性。

只要满足本文档定义的行为、文件系统结果、安全规则、退出码和必要性能边界，不同实现可以自由选择技术栈和内部设计。

## 目标

xdotter 是一个小型 dotfiles/环境配置部署工具。它的职责是从用户控制的 dotfile 仓库中的文件或目录，向目标位置（例如用户的 `$HOME`）创建和删除符号链接。

核心模型很简单：

```text
仓库中的源文件或目录  ->  目标路径上的符号链接
```

xdotter 不复制文件内容。它创建符号链接。

## 范围和非目标

xdotter 设计用于将用户自己的 dotfiles 部署到用户自己的环境中，例如 `$HOME`、`~/.config`、shell 配置文件、SSH 配置以及类似的用户可控路径。

xdotter 不是沙箱、权限边界、包管理器、备份系统，也不是多用户文件系统隔离工具。

当部署目标在部署过程中被不受信任的用户或进程并发修改时，xdotter 不保证行为安全。

用户应避免将链接部署到全局可写或其他不受信任的目录中。

## 支持平台

- Linux 是主要支持平台。
- macOS 是主要支持平台。
- Windows 是 best-effort 支持平台。

Windows 上的配置解析、依赖语义、规划语义、预演语义、冲突处理语义、命令交互、错误分类和退出码语义必须与 Linux/macOS 一致。

Windows 上的链接路径必须是 home 相对路径（`~/`）或盘符绝对路径。其他绝对路径形式（包括 UNC 路径、长路径前缀、POSIX 风格根路径、盘符相对路径）属于配置错误。

Windows 上路径语义的其他细节由实现决定，包括但不限于：`~/` 的具体展开方式、路径分隔符（`/` 与 `\`）的接受范围、大小写归一化规则、链接路径唯一性比较时的归一化策略。SPEC 不约束这些细节，跨实现的 Windows 路径行为不保证完全一致。

Windows 上的符号链接操作使用平台 API。符号链接创建可能因系统权限、开发者模式或系统策略失败；这类失败属于应用阶段错误。

Windows 上不执行 Unix mode 权限检查或修复。

## 配置格式

xdotter 使用固定名称的 TOML 配置文件：

```text
xdotter.toml
```

当前用户可见的受支持配置格式只有 TOML。

根配置必须是 TOML 表。

允许的顶层表只有：

- `[links]`
- `[dependencies]`

`[links]` 允许缺省，缺省时等同于空表。

`[dependencies]` 允许缺省，缺省时等同于空表。

空配置合法。对于没有任何链接的合法空配置，部署、卸载和状态命令必须成功且不处理任何链接。

未知顶层键或未知顶层表属于配置错误。

TOML 重复键按照 TOML 解析错误处理。

示例：

```toml
[links]
".zshrc" = "~/.zshrc"
".config/nvim" = "~/.config/nvim"

[dependencies]
"nvim" = "config/nvim"
```

`[links]` 和 `[dependencies]` 的 key/value 角色不同：

- `[links]` 的 key 是仓库内的源路径，value 是部署到的链接路径。
- `[dependencies]` 的 key 是依赖名（仅作标识符使用，不参与路径解析），value 是包含子 `xdotter.toml` 的相对子目录。

### `[links]`

`[links]` 表映射：

```text
source = link
```

- `source` 是要暴露的真实文件或目录。
- `link` 是要创建的符号链接路径。

`[links]` 中的键和值都必须是字符串。

多个源路径不得映射到同一个链接路径。链接路径在 `~/` 展开和路径规范化后重复，属于配置错误。

由于 `source` 是 TOML key，每个源路径只能映射到一个链接路径。这是有意的设计选择。如果需要将同一份内容暴露到多个位置，请在仓库中放置多份独立的源文件，或在配置文件本身（例如 shell rc）中使用 `source`/`include` 机制引用共享文件。

### `[dependencies]`

`[dependencies]` 表将依赖名称映射到相对子目录，这些子目录包含自己的 `xdotter.toml` 文件。

`[dependencies]` 中的键和值都必须是字符串。

## 配置信任模型

`xdotter.toml` 是部署配置，可能导致文件系统写入、创建符号链接、删除文件、删除目录和修改权限。

来自配置的路径必须在执行破坏性文件系统操作之前，按照其角色进行校验。

`--force` 不得跳过配置安全校验。

部署和卸载命令的 `--dry-run` 必须执行与对应真实命令相同的安全校验，尽管它不得修改文件系统。

`xd new --dry-run` 不得创建或修改 `xdotter.toml`。

xdotter 不支持跳过配置解析、配置校验或安全校验。

## 校验、规划和应用生命周期

xdotter 将 CLI 参数解析、配置校验、操作规划和文件系统应用分成不同阶段。

规划不是用户可选择的模式。规划是部署和卸载命令都会执行的内部阶段。

### CLI 参数解析阶段

xdotter 首先解析 CLI 参数，并确定：

- 当前命令是否有效
- 当前命令的参数是否有效
- 对定义冲突处理参数的命令，确定冲突处理模式：默认模式、强制模式或交互模式
- 对定义预演参数的命令，确定执行模式：应用模式或预演模式

如果出现 CLI 参数错误，xdotter 必须在读取配置之前失败。

### 配置加载和静态校验阶段

xdotter 加载根 `xdotter.toml`，并发现需要处理的依赖配置。

静态配置校验覆盖 TOML 解析、表结构、字段类型，以及不需要应用部署或卸载计划即可判断的路径规则。

### 规划阶段

规划阶段会解析路径、检查命令所需的文件系统状态、构建操作计划，并对该计划运行安全检查。

规划阶段不得修改文件系统。

规划阶段必须检查路径角色规则、依赖路径安全、符号链接拓扑、文件系统身份和权限安全。

配置错误和规划阻塞错误必须在应用任何文件系统修改之前被发现。

部署规划必须收集根配置和所有可达依赖配置中的配置错误和规划阻塞错误。

如果存在一个或多个配置错误或规划阻塞错误，xdotter 必须：

- 报告已发现的错误
- 以非零退出码结束
- 不创建、删除、替换或修改任何文件系统对象

可恢复冲突不是配置错误，也不是规划阻塞错误。可恢复冲突根据当前冲突处理模式生成对应计划，或跳过该链接并计为失败。

### 预演或应用

如果规划成功，xdotter 根据执行模式处理操作计划：

- 预演模式（`--dry-run`）：打印已验证的操作计划，然后退出，不修改文件系统。
- 应用模式：执行已验证的操作计划，并真正修改文件系统。

当前 CLI 模型中没有单独的部署检查命令。使用 `xd deploy --dry-run` 或 `xd undeploy --dry-run` 进行操作预检。

### 应用阶段错误

应用阶段仍然可能因为操作系统错误、权限不足、磁盘错误或文件系统状态变化而失败。

xdotter 不是事务系统。应用阶段中已经完成的操作不会因为后续失败而自动回滚。

如果某个应用阶段操作失败，xdotter 必须报告错误，并最终以非零退出码结束。

应用阶段一旦某个文件系统修改操作失败，xdotter 必须停止执行后续操作。

已经完成的操作不会自动回滚。

## 错误分类

xdotter 区分不同类型的失败情况。不同错误类型发生在不同阶段，并具有不同处理方式。

### CLI 参数错误

CLI 参数错误是命令行参数本身无效。

例如，同时指定互斥的 `--force` 和 `--interactive` 属于 CLI 参数错误。

CLI 参数错误发生在配置读取之前。出现 CLI 参数错误时，xdotter 必须退出，不得读取配置、检查文件系统或修改文件系统。

### 配置错误

配置错误是 `xdotter.toml` 内容违反配置格式或静态配置规则。

配置错误包括 TOML 语法错误、表结构错误、字段类型错误，以及能够仅根据配置内容判断的非法路径规则。

配置错误不能通过 `--force` 或 `--interactive` 修复。

以下情况属于配置错误：

- `xdotter.toml` 不是有效 TOML
- 根配置不是 TOML 表
- 存在未知顶层键或未知顶层表
- `[links]` 或 `[dependencies]` 的表结构无效
- `[links]` 或 `[dependencies]` 中的键或值类型无效
- 多个源路径映射到同一个链接路径
- 空路径字符串
- 源路径不是普通相对路径
- 源路径是 `.`
- 源路径包含父目录跳转组件 `..`
- 链接路径不是绝对路径，也不是 home 相对路径
- 链接路径在 `~` 展开后包含父目录跳转组件 `..`
- 链接路径解析为文件系统根目录或当前用户 home 目录本身（包括字符串结构上即可判断的 `~`、`~/`、`/`、仅由路径分隔符组成的字符串等情况）
- 依赖路径是绝对路径
- 依赖路径是 `.`
- 依赖路径包含父目录跳转组件 `..`
- 同一个 `[dependencies]` 表中，多个依赖项解析到同一个真实依赖目录
- 全局链接集合中，多个链接条目展开和规范化后的链接路径相同；该错误必须列出所有产生冲突的链接条目，包括各自所在的配置文件路径和声明的源路径

### 规划阻塞错误

规划阻塞错误是在操作规划阶段发现的、导致无法安全生成或应用操作计划的问题。

规划阻塞错误通常需要结合当前文件系统状态判断，例如源路径不存在、源路径或依赖目录解析为真实路径后逃出当前配置目录树、源路径和链接路径解析到同一个文件系统对象，或创建链接会产生无效符号链接拓扑。

规划阻塞错误不能通过 `--force` 或 `--interactive` 修复，也不能由用户确认后继续。

以下情况属于规划阻塞错误：

- 源路径不存在
- 源路径存在但不是普通文件或目录
- 源路径的任一组件是符号链接
- 源路径逃出当前配置目录树
- 依赖路径不存在
- 依赖路径存在但不是目录
- 依赖目录中不存在 `xdotter.toml`
- 依赖目录解析为真实路径后逃出当前配置目录树
- 依赖遍历检测到真实循环
- 链接路径或其父级违反"符号链接安全语义"小节定义的拓扑安全规则（包括源路径与链接路径解析到同一对象、链接路径位于源路径内部、源路径位于将被删除的已有真实链接目录内部、链接路径父级不是目录或是不安全的符号链接、创建符号链接会产生符号链接循环等）
- 权限修复需要修改的目标无法验证为预期源路径

### 可恢复冲突

可恢复冲突是能够在不违反配置规则、路径安全、依赖安全、符号链接拓扑安全、文件系统身份安全和权限安全的前提下被处理的问题。

可恢复冲突不是配置错误，也不是规划阻塞错误。

默认模式不处理可恢复冲突。强制模式自动处理可恢复冲突。交互模式在用户确认后处理可恢复冲突。

权限问题属于可恢复冲突。默认模式必须跳过该链接并计为失败；强制模式必须在安全验证通过后修复权限；交互模式必须询问用户是否修复权限。

### 应用阶段错误

应用阶段错误是在操作计划已经通过校验后，执行文件系统操作时发生的错误。

应用阶段错误包括创建父目录失败、删除冲突目标失败、创建符号链接失败、修改权限失败，或文件系统状态在规划后发生变化。

## 路径语义

### 路径形式

SPEC 区分三种路径形式：

- **绝对路径**：在 Linux/macOS 上以 `/` 开头；在 Windows 上是盘符绝对路径。
- **home 相对路径**：以 `~/` 开头；解析时，路径开头的 `~` 字符替换为当前用户 home 目录的路径，其后的 `/` 和剩余部分保持原样。
- **普通相对路径**：既不是绝对路径，也不是 home 相对路径，且不为空、不是 `.`。普通相对路径相对于声明它的配置目录解析。

不同位置（源路径、链接路径、依赖路径）允许的路径形式不同，由各自的小节规定。

### 配置目录

每个 `xdotter.toml` 都有一个配置目录。配置目录是该 `xdotter.toml` 所在目录。

相对源路径和依赖路径都相对于声明它们的配置目录解析，而不是相对于进程启动时的当前工作目录。

链接路径是部署目标，必须显式指向 home 目录下的位置或绝对位置，不支持普通相对路径。

每个 `xdotter.toml` 的**配置目录树**定义为该 `xdotter.toml` 所在目录及其所有子目录（不通过符号链接跳出该目录后到达的位置）。源路径和依赖路径解析后必须留在声明它们的 `xdotter.toml` 自己的配置目录树内，不得借 `..` 或符号链接进入兄弟目录、父目录或外部位置。每个依赖配置维持自己的独立配置目录树；依赖配置之间不共享同一棵根配置目录树。

### `~` 展开

只有以 `~/` 开头的路径会进行 home 展开。展开时，路径开头的 `~` 字符替换为当前用户 home 目录的路径；其后的 `/` 和剩余部分保持原样。

单独的 `~` 和 `~user` 形式不属于当前支持语义。

`~` 展开只发生在路径开头；路径中间的 `~` 不会展开。

如果需要展开 `~/` 但当前用户 home 目录无法确定，属于规划阻塞错误。命令必须在任何文件系统修改之前失败，并以非零退出码结束。

### 路径规范化

空路径字符串无效。

路径组件 `.` 允许被规范化消除。

重复路径分隔符允许被规范化合并。

尾随路径分隔符不具有额外语义。

### 源路径

源路径用于标识当前配置目录树内要暴露的真实文件或目录。

源路径必须是普通相对路径，并相对于声明它的配置目录解析。

源路径不得是绝对路径，也不得是 home 相对路径。

源路径不得包含父目录跳转组件 `..`。

源路径不得为空，也不得是 `.`。

源路径在部署前必须存在，并且必须指向普通文件或目录。

源路径的任何组件（包括最终组件和所有中间组件）都不得是符号链接。源路径不存在、不是普通文件或目录，或源路径的任一组件是符号链接，属于规划阻塞错误。

源路径必须仍然位于当前配置目录树内，并且不得产生无效的链接/源路径拓扑。由于源路径不允许包含符号链接组件，源路径解析为真实路径与按字面解释源路径必须等价。

### 链接路径

链接路径描述 xdotter 可以创建、删除或替换的文件系统位置。因此，链接路径是破坏性操作目标，比源路径受到更严格的处理。

规则：

- 链接路径必须是绝对路径或 home 相对路径。
- 链接路径不得是普通相对路径。
- 链接路径不得为空。
- 链接路径在 `~/` 展开之后，任何路径组件都不得包含父目录跳转（`..`）。
- 链接路径不得解析为文件系统根目录，也不得解析为当前用户 home 目录本身。这包括字符串结构上即可判断的情况：链接路径不得是 `~`、`~/`、`/`，也不得是仅由路径分隔符组成的字符串（例如 `//`）。这类静态可判断的非法情况不依赖 home 目录展开即可拒绝。
- 如果链接路径是普通相对路径或包含 `..`，属于配置错误，部署必须在任何文件系统修改之前失败。
- 无论是否使用 `--force`、`--interactive` 或 `--dry-run`，该规则都适用。

部署时，如果链接路径的父目录不存在，xdotter 允许创建缺失的父目录。

xdotter 不删除、替换或修复链接路径的父级文件系统对象。链接路径父级的安全规则（父级不是目录、父级是不安全的符号链接等）由"符号链接安全语义"小节统一定义。

理由：链接路径是破坏性操作目标。要求链接路径显式指向 home 目录或绝对位置，可以减少误解和误操作。允许普通相对路径或 `..` 会导致可见路径与实际操作位置不同，并使安全检查复杂化。

### 依赖路径

依赖路径用于标识当前配置目录树内包含自己 `xdotter.toml` 的子模块目录。

依赖路径必须是相对于当前配置目录的普通相对路径。

依赖路径不得是绝对路径。

依赖路径不得包含父目录跳转组件 `..`。

依赖路径不得为空，也不得是 `.`。

依赖路径的目标必须存在，必须是目录，并且必须包含自己的 `xdotter.toml`。

依赖目录解析为真实路径后，仍然必须位于当前配置目录树内。

如果依赖目录不存在、不是目录、缺少 `xdotter.toml`，或解析为真实路径后逃出当前配置目录树，部署必须在进入该依赖目录或修改文件系统之前失败。

### 文件系统身份

检查两个路径是否指向同一个文件系统对象时，xdotter 不得只依赖字符串相等。

对于已存在的路径，xdotter 必须比较解析后的文件系统身份。

具体的身份冲突拒绝规则（同对象、链接位于源内、源位于将删除目录内等）由"符号链接安全语义"小节统一定义。

## 符号链接安全语义

xdotter 必须防止无效的链接/源路径拓扑。

以下情况无效，即使使用 `--force` 也必须失败：

- 源路径和链接路径解析到同一路径
- 链接路径位于源路径内部
- 源路径位于一个将被删除的已有真实链接目录内部
- 链接路径任一已存在父级不是目录
- 链接路径任一已存在父级是符号链接，且该父级符号链接会导致实际创建位置等于源路径、位于源路径内部，或产生符号链接循环
- 创建该符号链接后，从链接路径跟随符号链接会回到链接路径自身或进入无限符号链接循环

xdotter 不自动修复或替换链接路径的任何父级文件系统对象。如果已有父级是普通文件、是不安全的符号链接，或会以其他方式导致不安全拓扑，规划必须失败关闭，而不是删除或替换该父级。

### 正确符号链接判定

正确符号链接由当前配置条目的预期源路径决定。

如果符号链接目标是相对路径，必须相对于链接路径的父目录解释。

如果符号链接目标存在，xdotter 必须比较该目标和预期源路径解析后的文件系统对象。

如果二者是同一个文件系统对象，该符号链接是正确符号链接。

如果目标存在但不是同一个文件系统对象，该符号链接是错误符号链接。

如果目标不存在，该符号链接是损坏符号链接。

已经存在且正确指向预期源路径的符号链接，必须视为成功并跳过。

## 执行模型

xdotter 将“冲突处理模式”和“执行模式”分开定义。

冲突处理模式决定 xdotter 如何处理链接路径上的可恢复冲突。

执行模式决定 xdotter 是否真正修改文件系统。

### 冲突处理模式

部署和卸载命令恰好有一种冲突处理模式处于启用状态：

- 默认模式
- 强制模式（`--force`）
- 交互模式（`--interactive`）

`--force` 和 `--interactive` 是可选的命令级操作参数。当用户没有指定 `--force` 或 `--interactive` 时，使用默认模式。

`--force` 和 `--interactive` 是互斥的冲突处理模式。如果用户同时指定二者，CLI 参数解析必须失败，命令不得进入配置读取、部署规划或文件系统修改阶段。

#### 默认模式

默认模式是保守模式。

默认模式处理以下情况：

- 创建不存在的链接
- 跳过已经正确部署的链接

默认模式不会自动替换链接路径上已有的普通文件、目录、错误符号链接或损坏符号链接。

如果链接路径存在可恢复冲突，默认模式必须跳过该链接并计为失败，最终导致命令以非零退出码结束。

#### 强制模式（`--force`）

强制模式会自动处理链接路径上的可恢复冲突。

强制模式在安全校验通过的前提下处理以下可恢复冲突：

- 替换链接路径上已有的普通文件
- 替换链接路径上已有的错误符号链接
- 替换链接路径上已有的损坏符号链接
- 替换链接路径上已有的空真实目录，前提是这样做不会删除或包含源路径

强制模式不能绕过配置校验、路径安全、依赖安全、符号链接拓扑安全、文件系统身份检查或权限安全检查。

xdotter 永远不递归删除真实目录。非空真实目录是可恢复冲突，但 xdotter 不自动处理；该链接必须跳过并计为失败。

#### 交互模式（`--interactive`）

交互模式会在处理可恢复的破坏性操作之前询问用户。

交互模式在用户确认后处理强制模式允许处理的同一类可恢复冲突。

交互模式不能让非法配置、不安全路径、无效符号链接拓扑或不安全依赖遍历变为合法。

对于不能由用户确认解决的问题，xdotter 必须直接报错，而不是询问用户是否继续。

交互模式按**每个破坏性操作**逐项询问，而非每条链接一次。如果同一条链接触发多个独立的破坏性操作（例如同时需要替换错误符号链接并修复源文件权限），用户必须分别为每个操作答 yes。任一必要操作被拒绝时，xdotter 必须跳过该链接并计为失败，不得继续执行该链接剩余的破坏性操作。

如果用户拒绝某个必要操作，xdotter 必须跳过该链接并计为失败，最终导致命令以非零退出码结束。

只有 `y` 或 `yes`（大小写不敏感）表示确认。

空输入、其他输入和 EOF 都表示拒绝。

如果需要确认时 stdin 不是 TTY，视为拒绝。

#### 模式关系

默认模式、强制模式和交互模式是互斥的冲突处理模式。

`--dry-run` 不属于冲突处理模式。对于定义了冲突处理模式的命令，`--dry-run` 允许和三种冲突处理模式中的任意一种组合使用。

### 执行模式

会修改文件系统的命令可以定义预演模式（`--dry-run`）。

部署和卸载命令都有两种执行模式：

- 应用模式：默认执行模式。通过校验的操作会真正应用到文件系统。
- 预演模式（`--dry-run`）：只进行加载、校验、规划和输出，不修改文件系统。

`xd new` 也支持预演模式。`xd new --dry-run` 只报告将创建模板配置文件，不实际创建文件。

`--dry-run` 不属于冲突处理模式。对于定义了冲突处理模式的命令，它允许和默认模式、强制模式或交互模式组合使用。

允许的部署和卸载组合包括：

```text
xd deploy
xd deploy --dry-run
xd deploy --force
xd deploy --force --dry-run
xd deploy --interactive
xd deploy --interactive --dry-run
xd undeploy
xd undeploy --dry-run
xd undeploy --force
xd undeploy --force --dry-run
xd undeploy --interactive
xd undeploy --interactive --dry-run
```

不允许的部署和卸载组合包括：

```text
xd deploy --force --interactive
xd deploy --force --interactive --dry-run
xd undeploy --force --interactive
xd undeploy --force --interactive --dry-run
```

在预演模式下，xdotter 必须执行与对应真实命令相同的配置加载、路径校验、依赖校验、符号链接拓扑检查、文件系统身份检查和权限安全检查。

无效配置和不安全操作计划在预演模式下必须失败。预演模式不得把无效或不安全操作描述成仿佛将会执行。

预演模式允许报告所选冲突处理模式下将要执行的操作，但不得：

- 创建文件或目录
- 删除文件、目录或符号链接
- 创建符号链接
- 修改权限
- 询问用户确认

`--interactive --dry-run` 对每个需要确认的可恢复冲突视为回答 no。

`--force --dry-run` 对每个强制模式可处理的可恢复冲突视为回答 yes。

### 竞态安全和失败关闭行为

xdotter 假设部署目标通常位于用户控制的目录中。

不支持由不受信任用户或进程对部署路径进行对抗性并发修改。

尽管如此，当在执行破坏性操作前无法验证目标状态时，xdotter 必须失败关闭，而不是执行可能不安全的操作。

在执行任何破坏性操作之前，xdotter 必须确认当前文件系统对象仍然满足该操作在规划阶段依赖的安全前提。

如果目标路径的类型、身份、符号链接目标、权限修复对象或目录内容已经不再匹配规划阶段验证过的状态，xdotter 必须失败关闭，并且不得执行该破坏性操作。

该规则不表示 xdotter 提供对抗性并发安全；它只要求在可观察到状态不匹配或无法验证时避免继续执行可能不安全的破坏性操作。

破坏性操作包括：

- 删除文件
- 删除目录
- 删除符号链接
- 替换已有路径
- 修改权限

应用阶段的状态再校验只在应用模式下执行。预演模式（`--dry-run`）只执行规划阶段校验，不模拟应用阶段的状态再校验；预演输出反映规划完成时刻观察到的状态。

## 输出语义

xdotter 将命令结果输出和日志/诊断输出分开处理。

命令结果输出是命令的主要结果，例如 `xd status` 的状态摘要、`xd deploy --dry-run` 的操作计划、`xd completion` 的补全脚本和 `xd version` 的版本号。命令结果必须输出到 stdout。

日志/诊断输出用于展示进度、普通操作信息、调试信息、警告和错误。

xdotter 使用可重复的 `-v/--verbose` 控制日志详细程度：

- 无 `-v`：输出命令结果、警告和错误
- `-v`：额外输出普通操作信息
- `-vv`：额外输出调试信息
- `-vvv`：额外输出跟踪级细节

超过三次 `-v` 时，行为等同于 `-vvv`。

各详细级别的输出应递增且不重复。`-v` 应比无 `-v` 提供更多的信息，
但不得将无 `-v` 时已报告的信息改用不同形式再次报告。
同一条信息只能通过一条输出路径出现一次。

多条错误同时输出时，每条错误行必须独立携带其类别标签。
错误聚合不得用一个外层标签覆盖内部各条错误的原始标签。

警告和错误必须输出到 stderr，且不得被隐藏。

普通进度信息、成功信息、调试信息和跟踪信息受 `-v` 级别控制。

xdotter 不提供完全静默模式。

输出文案不要求逐字稳定，但必须满足最低稳定契约：

- 命令结果输出到 stdout。
- 警告、错误和诊断输出到 stderr。
- 错误消息必须包含可识别的错误分类标识（CLI 参数错误、配置错误、规划阻塞错误、应用阶段错误之一）、相关配置文件路径或源/链接路径，以及简短原因。具体标签文案不强制要求逐字稳定，但实现应在自身文档中说明所用标签。
- 退出码、输出中的关键字段和关键字，以及命令产生的文件系统结果必须保持稳定。

## 性能边界

xdotter 的常见操作应适合交互式 CLI 使用；在普通本地 dotfiles 仓库规模下，非交互命令应保持秒级以内的响应体验。

实现不得假设配置规模很小。对于包含 O(10²-10³) 个链接条目和 O(10¹-10²) 个依赖配置的 dotfiles 仓库，配置加载、依赖发现、部署规划、卸载规划和状态检查应保持可用。

常见操作的工作量应大致随可达配置数量、链接条目数量和需要检查的文件系统路径数量增长。实现应避免会导致用户可感知退化的明显超线性行为，例如为每个链接重复完整遍历所有配置或依赖图。

性能要求只约束用户可感知的规模边界、响应体验和退化特征，不规定具体算法、数据结构、语言或库。

## 权限和敏感文件语义

xdotter 不禁止用户部署隐私文件或敏感文件。

xdotter 不读取被部署源文件的内容来推断类型、权限或安全性。xdotter 只解析和校验自己的配置文件内容。

如果链接路径匹配内置权限目标，xdotter 必须输出警告，提醒用户确认该文件确实由 xdotter 管理。该警告不阻止部署。

xdotter 只对已知会因权限过宽而被软件拒绝、忽略或强烈警告的目标路径执行内置权限检查。

权限分类只基于 `~/` 展开后的链接路径，而不基于源文件名、源文件内容或源路径类型。

当前内置权限检查只覆盖以下目标：

| 链接路径 | 期望权限 |
|---|---|
| `~/.ssh` | 不宽于 `0700` |
| `~/.ssh/config` | 不宽于 `0600` |
| `~/.ssh/authorized_keys` | 不宽于 `0600` |
| `~/.ssh/id_*`，且不以 `.pub` 结尾 | 不宽于 `0600` |
| `~/.ssh/*_rsa`、`~/.ssh/*_ed25519`、`~/.ssh/*_ecdsa`、`~/.ssh/*_dsa`，且不以 `.pub` 结尾 | 不宽于 `0600` |
| `~/.pgpass` | 不宽于 `0600` |
| `~/.netrc` | 不宽于 `0600` |
| `~/.gnupg` | 不宽于 `0700` |

未列入上表的路径不进行内置权限检查。

权限更严格时必须视为可接受。例如，期望不宽于 `0600` 的文件可以是 `0400`。

在 Unix 和 macOS 上，权限检查和修复使用平台文件 mode。修复权限的对象是预期源路径指向的真实文件系统对象，而不是符号链接自身。

在 Windows 上，xdotter 不执行 Unix mode 权限检查或修复。Windows 上的敏感文件警告仍可输出，但 Unix mode 权限问题不得计为失败。

如果内置权限目标存在权限问题，该权限问题属于可恢复冲突：

- 默认模式：跳过该链接并计为失败
- 强制模式（`--force`）：自动修复权限并继续部署
- 交互模式（`--interactive`）：询问用户是否修复权限；拒绝则跳过该链接并计为失败

修复权限之前，xdotter 必须验证要修改的目标是该链接对应的预期源路径。

如果目标无法验证为预期源路径，权限修复必须失败且不得修改权限。

在预演模式（`--dry-run`）下，xdotter 允许报告将要修复的权限，但不得实际修改权限。

普通 shell、editor、terminal 和 git 配置不进行内置权限检查。

## 依赖语义

部署一个配置时，必须递归部署 `[dependencies]` 下声明的依赖配置。

xdotter 必须先发现根配置和所有可达依赖配置，再构建全局计划。

配置发现、依赖处理和链接处理必须稳定有序。具体排序算法属于实现细节。

用户不得依赖 TOML 字段书写顺序影响部署结果。

依赖遍历必须根据依赖路径规则保持在当前配置目录树内。

部署和卸载在遍历依赖配置之前，必须应用相同的依赖路径校验规则。

依赖遍历必须检测真实循环。依赖图中的共享依赖不一定是循环。

同一个 `[dependencies]` 表中，多个依赖项不得解析到同一个真实依赖目录。

不同配置引用同一个真实依赖目录时，该共享依赖只处理一次。

所有可达配置中的链接条目组成全局链接集合。

全局链接集合中，展开和规范化后的链接路径必须唯一。

卸载行为必须与部署行为使用相同的依赖发现和校验语义。

## 命令

### 参数模型

xdotter 区分诊断参数和操作参数。

诊断参数只影响日志和诊断信息的详细程度，不得改变命令结果语义、文件系统结果、配置校验、安全校验、操作计划、错误分类或退出码语义。

`-v/--verbose` 是诊断参数，可用于所有命令，并可重复使用：

- 无 `-v`：输出命令结果、警告和错误
- `-v`：额外输出普通操作信息
- `-vv`：额外输出调试信息
- `-vvv`：额外输出跟踪级细节

超过三次 `-v` 时，行为等同于 `-vvv`。

操作参数会改变具体命令的执行语义，因此必须由具体命令定义。未在某个命令中定义的操作参数不得被该命令接受。

### 命令参数

| 命令 | 支持的操作参数 |
|---|---|
| `xd deploy` | `--dry-run`, `--force`, `--interactive` |
| `xd undeploy` | `--dry-run`, `--force`, `--interactive` |
| `xd status` | 无 |
| `xd new` | `--dry-run` |
| `xd completion <shell>` | 无 |
| `xd version` | 无 |

`--force` 和 `--interactive` 互斥。该互斥规则适用于定义了这两个参数的命令。

### 操作参数语义

`--dry-run` 表示预演对应命令的文件系统影响，不修改文件系统。

`--force` 表示自动处理该命令定义的可恢复冲突。它不得绕过配置校验、路径安全、依赖安全、符号链接拓扑安全、文件系统身份检查或权限安全检查。

`--interactive` 表示在处理该命令定义的可恢复破坏性操作前请求确认。它不得让非法配置、不安全路径、无效拓扑或不安全依赖遍历变为合法。

### 核心命令

`xd status` 和 `xd deploy --dry-run` 关注的问题不同：

- `xd status` 回答“当前环境是否已经按配置部署”。
- `xd deploy --dry-run` 回答“如果现在部署，会执行什么计划”。

二者都不得修改文件系统。

#### `xd deploy`

默认命令。读取 `xdotter.toml`，校验配置和路径安全，部署每个链接，并递归处理依赖。

`xd deploy` 支持 `--dry-run`、`--force` 和 `--interactive`。

详细部署行为由以下部分定义：

- 配置信任模型
- 路径语义（源路径、链接路径、依赖路径、身份）
- 符号链接安全语义
- 执行模型（冲突处理模式、执行模式）
- 权限和敏感文件语义
- 依赖语义

按链接路径状态划分的部署行为：

#### `xd undeploy`

读取 `xdotter.toml`，并删除与配置链接匹配且能够验证为当前配置条目的符号链接。

`xd undeploy` 支持 `--dry-run`、`--force` 和 `--interactive`。

卸载默认删除正确符号链接和损坏符号链接。错误符号链接属于可恢复冲突，而不是规划阻塞错误。

按链接路径状态划分的行为：

| 链接路径状态 | 默认模式 | 强制模式（`--force`） | 交互模式（`--interactive`） |
|---|---|---|---|
| 是正确符号链接 | 删除它 | 删除它 | 删除前询问；拒绝则跳过该链接并计为失败 |
| 是损坏符号链接 | 删除它 | 删除它 | 删除前询问；拒绝则跳过该链接并计为失败 |
| 是错误符号链接 | 可恢复冲突；跳过该链接并计为失败 | 删除它 | 询问后删除；拒绝则跳过该链接并计为失败 |
| 存在但不是符号链接 | 警告，计为失败，不删除 | 警告，计为失败，不删除 | 警告，计为失败，不删除 |
| 不存在 | 静默跳过，视为成功 | 静默跳过，视为成功 | 静默跳过，视为成功 |

卸载预演模式（`--dry-run`）使用同一张表构建和校验计划，但不会删除任何符号链接，也不会询问用户。

卸载在遍历依赖配置之前，必须应用与部署相同的依赖路径校验规则。

#### `xd status`

读取 `xdotter.toml`，并报告当前环境中每个已配置链接的部署状态。

`xd status` 要求配置本身合法。配置错误必须导致 status 失败，而不是产生状态报告。

`xd status` 检查当前文件系统状态，但不构建部署或卸载计划，也不修改文件系统。

`xd status` 不要求源路径存在。源路径不存在、不是普通文件或目录，或源路径任一组件是符号链接时，相关链接不得报告为已部署。

状态分类：

- **已部署**：链接路径是符号链接，且指向当前配置条目的预期源路径，且源路径存在并是普通文件或目录，且源路径任一组件均不是符号链接
- **错误链接**：链接路径是符号链接，目标存在，但不指向当前配置条目的预期源路径
- **损坏链接**：链接路径是符号链接，但目标不存在
- **源不存在**：链接路径是符号链接，目标指向当前配置条目的预期源路径，但源路径在文件系统上不存在
- **源类型错误**：链接路径是符号链接，目标指向当前配置条目的预期源路径，源路径存在，但源不是普通文件或目录、或源路径任一组件是符号链接
- **不是符号链接**：链接路径作为普通文件或目录存在（不是符号链接）
- **未部署**：链接路径不存在

status 必须检查已部署链接中已知敏感目标的权限，并报告任何权限问题。

如果存在未部署、错误链接、损坏链接、源不存在、源类型错误、非符号链接或权限问题，`xd status` 必须以非零退出码结束。

`xd status -v` 必须打印所有链接路径，包括正确的链接。

输出末尾的摘要格式：

```text
Status: N/M deployed
Not deployed: N
Wrong links: N
Broken links: N
Source missing: N
Source type invalid: N
Non-symlink paths: N
Permission issues: N
```

摘要字段语义：

- **M**：全局链接集合中（按"依赖语义"小节定义，所有可达配置去重后）的链接条目总数。
- **N**：状态为"已部署"的链接条目数（链接路径是符号链接、目标指向预期源、源合规）。已部署链接即便存在权限问题也计入 N；权限问题在 `Permission issues` 行独立报告。
- **Not deployed / Wrong links / Broken links / Source missing / Source type invalid / Non-symlink paths**：分别对应"状态分类"小节中同名状态的链接条目数。这六行的总和加上 N 必须等于 M。
- **Permission issues**：链接命中"权限和敏感文件语义"小节定义的内置权限目标且存在权限问题的链接条目数。该计数与上述其他行相互独立，可与"已部署"等任意状态共存。

### 辅助命令

#### `xd new`

在当前目录创建模板 `xdotter.toml`，其中包含注释掉的 `[links]` 和 `[dependencies]` 示例段落。

`xd new` 支持 `--dry-run`。`xd new --dry-run` 只报告将创建模板配置文件，不实际创建文件。

如果 `xdotter.toml` 已存在，必须报错失败。

#### `xd completion <shell>`

生成 shell 补全脚本。

支持的 shell：`bash`、`zsh`、`fish`。

补全脚本打印到 stdout，供用户 source 或重定向到补全目录。

#### `xd version`

将 xdotter 版本号打印到 stdout。

## 退出码

- `0`：成功 —— 所有操作均无错误完成
- 非 `0`：故障 —— 一个或多个操作失败；详细信息打印到 stderr。具体非零值由实现选择，脚本不应依赖某一特定非零值
